## 개요

이 시스템은 백준(Solved.ac) 문제 데이터를 **스마트하게** 동기화합니다.

### 🎯 핵심 특징
- ✅ **스마트 동기화**: 전체를 긁지 않고 신규 문제만 업데이트
- ✅ **청크 처리**: 100개씩 나눠서 효율적으로 크롤링
- ✅ **Rate Limit 준수**: 0.5초 대기로 API 제한 방지
- ✅ **Upsert**: 기존 데이터는 업데이트, 신규는 추가
- ✅ **백그라운드 작업**: 동기화 중에도 서버는 정상 운영

## API 사용법

### 1️⃣ 동기화 상태 조회

현재 DB와 Solved.ac의 상태를 비교합니다.

**Request:**
```bash
curl -X GET "http://localhost:8000/admin/sync/status"
```

**Response:**
```json
{
  "status": "success",
  "data": {
    "db_problem_count": 0,
    "db_max_id": 0,
    "solved_max_id": 35000,
    "need_sync": true,
    "pending_problems": 35000
  }
}
```

**필드 설명:**
- `db_problem_count`: 현재 DB에 저장된 문제 수
- `db_max_id`: DB에 있는 가장 큰 문제 번호
- `solved_max_id`: Solved.ac의 최신 문제 번호
- `need_sync`: 동기화가 필요한지 여부
- `pending_problems`: 동기화 대기 중인 문제 수

---

### 2️⃣ 문제 동기화 실행

백그라운드에서 동기화를 시작합니다.

**Request:**
```bash
curl -X POST "http://localhost:8000/admin/sync/problems"
```

**Response:**
```json
{
  "status": "started",
  "message": "동기화 작업이 백그라운드에서 시작되었습니다.",
  "detail": "작업 완료까지 시간이 걸릴 수 있습니다. 서버 로그를 확인하세요."
}
```

## 주의사항

### ⚠️  Rate Limit
- Solved.ac API는 요청 제한이 있습니다.
- 시스템은 자동으로 0.5초씩 대기하지만, 과도한 호출은 피하세요.

### ⏱️  실행 시간
- 전체 동기화(1000번~35000번)는 약 **4~5시간** 소요됩니다.
- 계산: 34000개 ÷ 100 = 340번 호출 × 0.5초 ≈ 170초 (API 호출만)
- 실제로는 DB 저장 시간 포함하여 더 오래 걸릴 수 있습니다.

### 🔄 재실행 안전성
- **안전합니다!** 이미 저장된 문제는 스킵하고 신규만 추가합니다.
- 실수로 여러 번 실행해도 중복 저장되지 않습니다.

### 📊 스마트 동기화의 장점
- **초기 실행**: 1000~35000 (34000개) → 긴 시간 소요
- **다음 실행**: 35001~35050 (50개만) → 1분 이내 완료 ⚡

## 트러블슈팅

### 문제: "동기화 대상: 0개"라고 나옵니다
**원인**: 이미 최신 상태입니다.
**해결**: 정상입니다. 새 문제가 추가되면 자동으로 감지됩니다.

### 문제: API 호출이 실패합니다
**원인**: Solved.ac API가 응답하지 않거나 Rate Limit에 걸렸습니다.
**해결**: 잠시 후 다시 시도하세요.

### 문제: 동기화가 중간에 멈췄습니다
**원인**: 서버 재시작 또는 네트워크 문제
**해결**: 다시 `POST /admin/sync/problems` 호출하면 중단된 지점부터 재개됩니다.
